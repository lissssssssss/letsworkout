# iOS 独立 App 抖音直播动作跟练比对 完整技术方案

## 一、产品定位与核心形态

用户正常使用抖音 App 全屏观看主播直播 / 健身短视频，无需在自研 App 内嵌入抖音、无需逆向抖音直播流、无需网络抓包。

自研独立 iOS App 通过苹果官方系统能力全局录屏捕获手机屏幕画面，同时调用前置摄像头采集用户实时动作；全程端侧离线 AI 完成双路人体骨架提取、动作姿态相似度比对、实时打分与动作纠错。

- **核心体验**：用户无任何操作割裂感，正常刷抖音即可跟练
- **运行模式**：全本地离线，不上传画面、不上传摄像头数据
- **分发合规**：可上架 App Store（存在审核风险，见第七节应对策略）

## 二、整体业务流程

1. 用户安装并打开自研 iOS App
2. 主动授权：摄像头权限、屏幕录制权限（iOS 系统官方弹窗）
3. 开启「开始跟练」，唤起系统录屏授权，启动 ReplayKit Broadcast Upload Extension
4. 用户切回抖音 App，任意进入直播间 / 健身短视频播放
5. 自研 App 以画中画悬浮窗（展示实时评分 + 摄像头预览画面）驻留前台
6. 实时并行处理：
   - Extension 捕获屏幕帧 → 通过 IPC 传输至主 App → 识别主播人体骨架
   - 主 App 前置摄像头实时取帧 → 识别用户人体骨架
   - 本地算法实时计算动作相似度，输出 0~100 分、动作标准 / 错误提示
7. 用户可随时切回 App 查看跟练记录、动作复盘；手动结束即关闭录屏与摄像头

### Extension → 主 App 帧传输方案（IPC）

这是整个系统的核心链路，Extension 进程内存仅 50MB，**不能在 Extension 内运行 AI 推理**，必须高效传帧到主 App。

**方案：App Groups 共享内存 + 环形缓冲区 + Darwin Notification 信号**

| 组件 | 说明 |
|------|------|
| 共享容器 | App Groups (`group.com.app.screencast`) 提供跨进程共享文件 / 内存空间 |
| 帧缓冲区 | `mmap` 映射共享内存，环形缓冲区容纳 3 帧（避免读写冲突） |
| 帧格式 | Extension 内将 CMSampleBuffer 缩放至 480×640，转 BGRA raw 或 JPEG 压缩（~100KB/帧） |
| 同步机制 | Darwin Notification (`CFNotificationCenterGetDarwinNotifyCenter`) 通知主 App 新帧就绪 |
| 轮询兜底 | 主 App 侧 `DispatchSource.makeTimerSource` 每 33ms 轮询共享内存 write index，防止 Darwin Notification 高频下合并丢失 |
| 背压控制 | 环形缓冲写满时 Extension 丢弃最旧帧（容忍丢帧，优先实时性） |
| 容错处理 | 主 App 每 500ms 心跳检测 Extension 存活状态；Extension 被杀后自动提示用户重新开启 |

**帧传输延迟预算**：

```
Extension 收帧 → 缩放压缩（~5ms）→ 写入共享内存（~1ms）→ Darwin Notify（~1ms）→ 主 App 读取解码（~3ms）
总 IPC 延迟：~10ms
```

**Extension 内存控制策略**：

- 不持有超过 1 帧的 CMSampleBuffer 引用
- 缩放使用 `vImage`（Accelerate 框架），避免额外内存分配
- 内存峰值控制在 30MB 以内，预留安全余量

## 四、依赖 iOS 系统核心能力

### 1. ReplayKit + Broadcast Upload Extension

- **能力**：苹果官方全局屏幕录制扩展（iOS 12+）
- **作用**：捕获手机任意 App 全屏实时视频流（抖音、其他应用均可）
- **规则**：必须用户主动授权，系统状态栏显示橙色录制指示器，无法静默录屏
- **限制**：Extension 内存硬限 50MB，超限即被系统 SIGKILL

### 2. AVFoundation 前置摄像头

- 主 App 进程内通过 `AVCaptureSession` 采集前置摄像头
- 输出格式：`kCVPixelFormatType_32BGRA`，分辨率 480×640，帧率 30FPS
- 与 ReplayKit 互不冲突（Extension 录屏不包含摄像头画面）

### 3. 画中画悬浮窗（合规保活）

**实现方式**：使用 `AVPictureInPictureController` + `AVSampleBufferDisplayLayer`

- PiP 窗口内容：实时渲染摄像头预览画面 + 叠加评分 UI
- 这是**真实的视频播放**（摄像头帧持续输入 DisplayLayer），非空壳保活
- 用户可通过 PiP 窗口实时看到自己的动作和评分反馈
- PiP 窗口最小尺寸约 150×100pt，足够展示评分数字 + 缩略画面

**保活原理**：系统认定 App 正在进行视频播放，不冻结进程

### 4. 后台模式配置

```xml
<!-- Info.plist -->
<key>UIBackgroundModes</key>
<array>
    <string>audio</string>          <!-- 语音反馈播报 -->
    <string>processing</string>     <!-- BGProcessingTask -->
</array>
```

- `audio` 模式：用于语音动作纠错播报，同时辅助保活
- `processing` 模式：配合 `BGProcessingTaskRequest` 处理后台数据整理
- 主保活手段是 PiP，后台模式为辅助补充

## 五、端侧 AI 整体方案

### 1. 模型与框架选型

| 项目 | 说明 |
|------|------|
| 姿态检测框架 | MediaPipe Pose Landmark（iOS Swift Task API） |
| 模型规格 | Lite 版本（默认），Full 版本仅高端机启用 |
| 模型能力 | 输出单人 33 个人体骨骼关键点（x, y, z 坐标 + visibility 置信度） |
| 推理后端 | Core ML delegate（优先）→ Metal GPU delegate（回退） |
| 模型体积 | Lite ~3MB，Full ~6MB，打包进 App Bundle |

### 2. 双路推理调度策略

由于两路 MediaPipe 实例共享 GPU 资源，需要精细调度：

**交错推理模式**（默认）：

```
时间轴：|---帧1---|---帧2---|---帧3---|---帧4---|
屏幕路：  [推理]            [推理]
摄像头路：          [推理]            [推理]
有效帧率：每路 15FPS，总 GPU 负载等效单路 30FPS
```

**并行推理模式**（A15+ 芯片，16 核 GPU）：

```
两路同时推理，各 20~25FPS
需监控 GPU 占用率，超 80% 自动降级为交错模式
```

**性能基线实测数据**（需 POC 验证，以下为预估）：

| 机型 | 芯片 | 单路推理耗时 | 双路有效帧率 | 模式 |
|------|------|-------------|-------------|------|
| iPhone 15 Pro | A17 Pro | ~15ms | 25FPS×2 | 并行 |
| iPhone 13 | A15 | ~22ms | 20FPS×2 | 并行 |
| iPhone 11 | A13 | ~35ms | 15FPS×2 | 交错 |
| iPhone SE 3 | A15 | ~22ms | 15FPS×2 | 交错（散热受限） |

> **注意**：以上数据为理论预估，需在 POC 阶段通过 Instruments 实测验证。最低支持机型暂定 iPhone 11（A13），低于此机型不保证体验。

### 3. 屏幕画面 ROI 提取（主播区域检测）

抖音直播画面包含大量 UI 元素（弹幕、礼物特效、按钮），需准确提取主播人体区域：

**策略一：MediaPipe Pose 自带人体检测**

- MediaPipe Pose 内置 Person Detector，首帧全图检测人体 bounding box
- 后续帧在上一帧检测区域附近 tracking，无需每帧全图扫描
- 检测到多人时取面积最大的 bounding box（主播通常占画面主体）

**策略二：固定 ROI + 动态微调**

- 抖音直播画面布局相对固定：主播在画面中央偏上 60% 区域
- 初始 ROI 取画面中央 70% 区域，排除底部弹幕和顶部状态栏
- 配合 Person Detector 结果动态调整 ROI 边界

**多人 / 特殊场景处理**：

- 多人连麦：仅取最大人体区域（画面占比最大者为主播）
- 无人检测到：暂停评分，UI 提示"未检测到主播动作"
- 画面切换（进入商品页等）：连续 N 帧无人体 → 自动暂停，恢复后继续

### 4. 动作比对核心算法设计

- **骨架归一化**：以髋部中心为原点，肩宽为尺度因子，将 33 点坐标归一化到 [-1, 1] 空间，消除距离 / 身高 / 画面裁剪差异
- **镜像自适应校正**：检测前置摄像头标志位，自动对所有 x 坐标取镜像（`x' = 1 - x`），对齐主播正面视角
- **核心关节角度计算**：提取 8 组关键夹角（左右肩、左右肘、左右髋、左右膝），使用向量点积计算角度，舍弃手指 / 面部等末端细节
- **帧时间戳归一化**：屏幕帧（来自 Extension IPC）与摄像头帧时间基线不同，进入 DTW 前统一转换为相对时间戳（以各自首帧为 t=0），确保时序对齐正确性
- **DTW 动态时间规整**：滑动窗口 DTW（窗口 2 秒 / 30~60 帧），柔性对齐用户与主播动作时间轴，约束带宽 ±10 帧防止过度拉伸
- **DTW 冷启动处理**：跟练开始前 2 秒 DTW 窗口未填满期间，退化为逐帧关节角度对比模式（不做时序对齐），待窗口填满后平滑切换至完整 DTW
- **加权相似度打分**：关节角度误差（权重 0.6）+ 归一化骨骼点余弦距离（权重 0.4），输出 0~100 分
- **时序滤波平滑**：指数移动平均（EMA，α=0.3），抑制单帧抖动，保持评分连续性
- **低置信度关键点过滤**：visibility < 0.5 的关键点不参与计算，相关关节角度使用上一帧有效值

## 七、合规与 App Store 上架策略

### 合规基础

- 不逆向抖音协议、不破解直播流、不抓包接口，纯使用苹果官方 ReplayKit API
- 所有视频帧、摄像头数据、AI 推理全部本地离线运算，无任何数据上传
- 摄像头、屏幕录制权限均由用户主动授权，隐私说明文案清晰

### 审核风险点与应对

| 风险点 | Apple 条款 | 应对策略 |
|--------|-----------|----------|
| 录制第三方 App 内容 | 2.5.1 不得滥用系统 API | 强调"用户主动发起，仅本地 AI 处理，不存储不上传录制画面"；App 描述定位为"AI 健身教练"而非"录屏工具" |
| PiP 非传统视频播放 | PiP 需真实视频内容 | PiP 内播放摄像头实时画面 + 评分叠加，属于合法视频内容展示 |
| 后台长时间运行 | 后台执行需合理理由 | 使用 audio 模式（语音反馈）为合法后台理由，实际也确实有语音播报 |
| 与抖音功能竞争 | 无直接条款，但可能被拒 | App 定位为通用"运动跟练工具"，不在描述中提及抖音；支持任意屏幕内容 |

### 被拒后的降级方案

如果 ReplayKit 录屏方案被 Apple 审核拒绝，准备以下降级路径：

1. **方案 B：用户手动录制视频导入**
   - 用户先在抖音录屏保存到相册，再导入 App 做离线比对
   - 体验降级（非实时），但完全无审核风险

2. **方案 C：双手机模式**
   - 一台手机播放抖音，另一台手机摄像头同时拍摄屏幕和用户
   - 单摄像头画面内做双人姿态检测
   - 体验降级较大，但零审核风险

3. **方案 D：内置课程内容**
   - App 内置自有版权的健身教练视频
   - 用户对着 App 内视频跟练，无需录屏
   - 体验最好但需要内容制作成本

## 八、核心技术难点与解决方案

| 难点 | 解决方案 | 验证方式 |
|------|----------|----------|
| Extension 50MB 内存限制 | Extension 内仅做帧缩放 + IPC 转发，AI 推理全在主 App | Instruments Memory Graph 验证峰值 < 35MB |
| Extension 被系统杀死 | 主 App 心跳检测 + 用户提示重新开启；环形缓冲防数据丢失 | 压力测试（后台 10min+ 存活率） |
| 双路推理 GPU 争抢 | 交错推理调度 + 动态降级策略；A15 以下芯片自动交错模式 | GPU Timeline (Instruments) 验证无 stall |
| 抖音画面 UI 干扰识别 | MediaPipe 内置 Person Detector + 动态 ROI + 最大人体过滤 | 准备 50+ 抖音直播截图做检测率测试 |
| 自拍镜像视角不匹配 | 前置摄像头坐标自动 x 轴镜像翻转 | 单元测试验证左右对称动作评分一致 |
| 动作节奏不一致 | 滑动窗口 DTW（2s 窗口，±10 帧约束） | 用不同速度录制同一动作，验证评分稳定性 |
| 高低配 iPhone 性能差异 | 启动时 benchmark 检测 → 自动选择推理模式和帧率 | 覆盖 A13~A17 Pro 全机型帧率测试 |
| 发热导致性能下降 | 监控 `ProcessInfo.thermalState`，过热时降帧至 10FPS + 提示用户休息 | 持续 30min 运行散热测试 |
| 电量消耗过快 | 目标：满电运行 > 2小时；自适应帧率 + 低功耗推理模式 | 实测不同机型每小时耗电量 |

## 九、错误恢复与边界处理

| 场景 | 处理策略 |
|------|----------|
| Extension 进程被杀 | 主 App 检测 IPC 超时 → 暂停评分 → 提示用户重新开启录屏 |
| 摄像头被其他 App 抢占 | `AVCaptureSession` interrupted 通知 → 暂停用户骨架检测 → 提示 |
| 用户手动关闭 PiP 窗口 | 检测 `pictureInPictureControllerDidStopPictureInPicture` → 自动暂停跟练 → 全屏 Toast 提示"画中画已关闭，点击恢复跟练" → 用户点击后重新激活 PiP |
| 系统来电 / 通知打断 | 暂停推理 → 结束后自动恢复 → 中断时段不计分 |
| 内存警告 | 清理帧缓冲区 → 降低分辨率至 360×480 → 极端情况优雅退出 |
| 屏幕录制权限被收回 | 检测 `RPBroadcastSampleHandler.finishBroadcast` → 结束本次跟练 |
| 连续 5s 无有效人体检测 | 暂停评分计算 → UI 提示"请确保全身在画面内" |
| App 从后台恢复 | 检查 Extension 存活 → 恢复摄像头 Session → 重新同步时间基线 |
| 后台摄像头 Session 中断 | PiP 活跃期间摄像头 Session 可维持；若 PiP 被关闭导致 App 冻结，恢复时需重建 `AVCaptureSession` 并重新 startRunning |

## 十、能力指标与设备要求

### 运行指标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 端到端延迟 | < 150ms | 屏幕帧到评分输出全链路（IPC 10ms + 推理 30ms + 算法 10ms + 渲染 16ms） |
| 双路有效帧率 | 15~25 FPS | 因机型而异，最低保证 15FPS |
| 评分准确率 | 85%~95% | 标准健身动作（深蹲、开合跳等）；复杂舞蹈约 75%~85% |
| 连续运行时长 | > 60 分钟 | 不因发热或内存泄漏崩溃 |
| 电量消耗 | < 40%/小时（高性能）/ < 25%/小时（省电） | 高性能：双路满帧推理；省电：交错模式 + 降帧至 10FPS |
| Extension 存活率 | > 99% | 正常使用 30 分钟内不被系统杀死 |

### 设备要求

| 等级 | 机型 | 体验 |
|------|------|------|
| 最佳体验 | iPhone 13 Pro 及以上（A15 Pro+） | 并行双路 25FPS，Full 模型 |
| 标准体验 | iPhone 11 / 12 / SE3（A13~A15） | 交错模式 15FPS，Lite 模型 |
| 不支持 | iPhone XS 及以下（A12-） | 性能不足，App 内提示不兼容 |
